# Claude Code لديه ميزة Workflows. ماذا عن النماذج الأخرى؟ جربت OpenWorkflows

كنت أتابع عن كثب ميزة workflows في Claude Code مؤخراً.

ما يثير اهتمامي ليس مجرد "ميزة أخرى". إنها تخرج العمل المعقد من دورة محادثة تلو الأخرى. يمكن تقسيم المهام إلى وكلاء فرعيين وفروع متوازية وخطوط أنابيب، ثم تنسيقها بواسطة البرامج النصية.

هذا مهم لأن سير العمل لم يعد ترتيباً مؤقتاً داخل محادثة واحدة. يصبح شيئاً يمكنك حفظه وتحريره وإعادة استخدامه.

كما أثار ذلك سؤالاً لدي: إذا كانت مهام سير العمل أصبحت طبقة شائعة في البرمجة بالذكاء الاصطناعي، فلماذا ترتبط بنموذج واحد أو CLI واحد؟

لذا جربت OpenWorkflows. إنه يحول مهام سير العمل بأسلوب Claude Code إلى لوحة مرئية، ويحاول جعل نفس سير العمل يستهدف Claude Code وCodex وGemini والمزيد من بيئات التشغيل المحلية أو السحابية.

لا يبدأ هذا البرنامج التعليمي بمفاهيم مجردة. إنه يمشي عبر لقطات الشاشة بالترتيب. المثال ملموس: جعل OpenWorkflows يدعم سمات مظهر متعددة، مع جعل Pencil هو الافتراضي، والسماح بالتبديل في الإعدادات / المظهر.

> هذا هو البرنامج التعليمي للاستخدام المعتمد على لقطات الشاشة.
>
> النسخة الإنجليزية: [Claude Code Has Workflows. What About Other Models? I Tried OpenWorkflows](claude-code-workflow-openworkflow.en.md)

## 0. ابدأ بالواجهة النهائية

<p align="center">
  <img src="images/0-标题使用.png" alt="لوحة OpenWorkflows وسجل التاريخ وخصائص العقد ومنطقة إدخال AI" width="960">
</p>
<p align="center"><em>الشكل 0: مساحة عمل OpenWorkflows الرئيسية، مع المخطط في المركز وخصائص العقد على اليمين وإدخال وإخراج AI في الأسفل.</em></p>

تحتوي واجهة المستخدم الرئيسية على أربعة أجزاء: سجل سير العمل على اليسار، واللوحة المرئية في المركز، وخصائص العقد والمطالبات الشائعة على اليمين، ولوحات إدخال AI والاستجابة في الأسفل.

يحمل سير العمل في لقطة الشاشة عنوان "خطة مظهر Pencil متعدد السمات". إنه ليس رسماً تخطيطياً ثابتاً. إنه سير عمل يمكنك الاستمرار في تحريره وتحويله إلى برنامج نصي وتشغيله وإعادة زيارته لاحقاً.

## 1. تنزيل OpenWorkflows

<p align="center">
  <img src="images/1-下载.png" alt="صفحة OpenWorkflows على GitHub مع مدخل الإصدارات" width="840">
</p>
<p align="center"><em>الشكل 1: ابحث عن أحدث إصدار من قسم Releases في صفحة GitHub.</em></p>

أسرع طريقة للتجربة هي فتح صفحة مشروع GitHub وتنزيل أحدث إصدار من Releases.

تجعل لوحة About على اليمين الموقع واضحاً. هذا محرر مرئي يوسع مهام سير عمل Claude Code إلى Codex وGemini والمزيد من بيئات تشغيل LLM.

إذا كنت تريد العمل على الكود، فاستنسخ المستودع وابدأ من دليل `app/`:

```bash
npm install
npm run dev
```

لتطبيق سطح المكتب:

```bash
npm run desktop
```

## 2. تأكيد الإعدادات العامة ومدخل التشغيل

<p align="center">
  <img src="images/2-通用设置.png" alt="صفحة الإعدادات العامة في OpenWorkflows" width="640">
</p>
<p align="center"><em>الشكل 2: قم بتكوين اللغة وCLI المحلي وShell التشغيل في الإعدادات / عام؛ واختر نموذج / قناة التشغيل النشطة أسفل إدخال AI.</em></p>

قبل رسم أي شيء، افتح **الإعدادات / عام**. هنا يمكنك تكوين لغة الواجهة، والترجمة التلقائية للمطالبات، وCLI المحلي، وShell التشغيل.

تمت إزالة تبويب **النماذج** القديم. لم يعد النموذج أو القناة النشطة يضبطان من صفحة الإعدادات؛ اخترهما للطلب الحالي من قائمة runtime أسفل إدخال AI.

إذا احتاجت عقدة محددة إلى نموذج مختلف، فحدد العقدة وغيّر النموذج من خصائص العقدة. إذا تركته فارغاً، فسترث العقدة اختيار workflow أو الاختيار العام.

## 3. إنشاء سير عمل جديد وإدخال طلب

<p align="center">
  <img src="images/3-新建workflow.png" alt="إنشاء سير عمل جديد وإدخال طلب في إدخال AI" width="840">
</p>
<p align="center"><em>الشكل 3: انقر على سير عمل جديد، ثم صف سير العمل في إدخال AI في الأسفل يميناً.</em></p>

بعد تأكيد الإعدادات العامة ومدخل التشغيل، انقر على **سير عمل جديد** على اليسار. تبدأ اللوحة بهيكل أدنى: Start وAgent واحد وEnd.

نقطة البداية الحقيقية ليست الرسم اليدوي للعقد. إنها إدخال AI في الزاوية السفلية اليمنى. في هذا المثال، كتبت:

```text
أريد أن يدعم OpenWorkflows سمات مظهر متعددة،
وأن يكون Pencil هو الافتراضي،
والسماح بالتبديل بينها في الإعدادات / المظهر.
```

بعد إرساله بـ `Ctrl+Enter`، أو بالنقر على زر الإرسال، يحول OpenWorkflows الطلب إلى مخطط سير عمل قابل للتحرير.

## 4-1. إنشاء مخطط سير العمل

<p align="center">
  <img src="images/4-1生成workflow蓝图.png" alt="مخطط سير العمل المنشأ من الطلب" width="960">
</p>
<p align="center"><em>الشكل 4-1: يقسم AI الهدف إلى فروع متوازية وعقدة ملخص وعقد تنفيذ وتحقق وتسليم.</em></p>

بمجرد إرسال الطلب، يعيد OpenWorkflows كتابة الخطوة الحالية إلى سير عمل كامل.

يصبح المخطط في لقطة الشاشة تقريباً:

```text
Start
  -> استكشاف دعم المظهر بالتوازي
      -> البحث في نقاط الدخول الحالية
      -> تصميم نظام السمات
      -> تصميم سمة Pencil الافتراضية
  -> تلخيص خطة التنفيذ
  -> تنفيذ المظهر متعدد السمات
  -> ربط تبديل الإعدادات / المظهر
  -> التحقق والمراجعة
  -> تسجيل نتائج التسليم
  -> End
```

المهم هنا ليس مدى جمال الرسم البياني. إنه أن الهدف الغامض يصبح خطة قابلة للتنفيذ.

لا تزال لوحة خصائص العقد على اليمين تسمح لك بفحص التسميات والأنواع والفروع وأنواع الوكلاء والمخططات. التوليد لا يمنعك من تحرير الهيكل.

## 4-2. عرض البرنامج النصي المنشأ

<p align="center">
  <img src="images/4-2蓝图脚本.png" alt="مربع حوار البرنامج النصي لسير العمل المنشأ" width="960">
</p>
<p align="center"><em>الشكل 4-2: استخدم زر البرنامج النصي لفحص برنامج سير العمل المنشأ من اللوحة.</em></p>

يوجد زر **Script** في الشريط العلوي. افتحه وسترى البرنامج النصي المنشأ من المخطط الحالي.

في لقطة الشاشة، يمكنك رؤية هياكل `parallel(...)` و`agent(...)`. تصبح العقد المتوازية فروعاً متزامنة، وتصبح العقد العادية استدعاءات وكيل فردية.

هذا يظهر أن OpenWorkflows ليس مجرد رسم مربعات. اللوحة مدعومة بهيكل سير عمل مشترك يمكنه لاحقاً استهداف بيئات تشغيل مختلفة.

## 5. استمر في التحسين بالمطالبات الشائعة

<p align="center">
  <img src="images/5-使用常用提示词.png" alt="لوحة المطالبات الشائعة ومنطقة إدخال AI" width="960">
</p>
<p align="center"><em>الشكل 5: تدفع المطالبات الشائعة تعديلات سير العمل النموذجية إلى منطقة إدخال AI.</em></p>

بعد إنشاء المخطط، ليس عليك تشغيله فوراً. لوحة **المطالبات الشائعة** على اليمين أفضل لتحسين التدفق.

يتم تجميع المطالبات حسب السيناريو: التوضيح والوضوح والاكتمال والتكلفة والهيكل والموثوقية والأداء والتوازي والتحقق.

تظهر لقطة الشاشة مطالبة **توضيح الطلب**. إنها تملأ إدخال AI بطلب لتأكيد الغموض الرئيسي قبل تغيير الرسم البياني.

هذا مفيد لأن العديد من إخفاقات سير العمل ليست إخفاقات نموذجية. إنها تحدث لأن الهدف أو الحدود أو مسارات الفشل أو استراتيجية التكلفة لم يتم ذكرها بوضوح كافٍ.

## 6. تأكيد الحدود بالخيارات التفاعلية

<p align="center">
  <img src="images/6-交互选择.png" alt="أزرار الخيارات التفاعلية في استجابة AI" width="640">
</p>
<p align="center"><em>الشكل 6: عندما يكون الطلب غامضاً، يقدم AI خيارات حتى تتمكن من تأكيد النطاق أولاً.</em></p>

بعد اختيار **توضيح الطلب**، لا يغير AI الرسم البياني فوراً. بدلاً من ذلك، يسأل سؤال متابعة: إلى أي مدى يجب أن يذهب تبديل السمات؟

تقدم لقطة الشاشة خيارين: فقط شحن سمة Pencil الافتراضية وترك هيكل التوسع في مكانه، أو شحن Pencil بالإضافة إلى سمات متعددة قابلة للتبديل.

بمجرد أن تختار، يكتب AI هذا القرار مرة أخرى في مخطط سير العمل ويخرج IRGraph المحدث. هذا يقلل من خطر أخذ AI لسير العمل في الاتجاه الخاطئ بمفرده.

## 7. انقر على تشغيل

<p align="center">
  <img src="images/7-运行.png" alt="زر التشغيل العلوي في OpenWorkflows" width="960">
</p>
<p align="center"><em>الشكل 7: بعد أن يصبح المخطط جاهزاً، انقر على تشغيل في الشريط العلوي.</em></p>

بعد تأكيد الهيكل واختيار runtime والحدود الرئيسية، انقر على **تشغيل**.

من الأفضل عدم تشغيل سير العمل لحظة إنشائه. تحقق أولاً مما إذا كانت الفروع المتوازية منطقية، وما إذا كانت عقدة الملخص تأتي بعد الفروع، وما إذا كان التحقق يغطي النتيجة النهائية.

إذا كانت العقدة غير واضحة فقط في المسؤولية، يمكنك تحرير تسميتها ومطالبتها ونوع الوكيل أو المخطط قبل التشغيل مرة أخرى.

## 8. راقب حالة التشغيل

<p align="center">
  <img src="images/8-运行中.png" alt="حالة التشغيل مع زر الإيقاف" width="960">
</p>
<p align="center"><em>الشكل 8: أثناء التشغيل، يتغير الزر إلى "جارٍ التشغيل... إيقاف"، وتعرض كل عقدة حالتها.</em></p>

عندما يبدأ سير العمل، يتغير الزر العلوي إلى **جارٍ التشغيل... إيقاف**. يتم قفل إدخال AI في الأسفل حتى لا يتغير المخطط أثناء التشغيل.

تعرض اللوحة حالة العقد مباشرة. في لقطة الشاشة، اكتملت Start، ولا تزال العقدة المتوازية قيد التشغيل، ويظهر العداد العلوي الأيمن تقدم التنفيذ.

هذا أكثر قابلية للقراءة من سجل طويل. إذا فشل شيء ما، لا تحتاج إلى التخلص من المطالبة بأكملها. يمكنك العثور على العقدة الفاشلة وتعديل مطالبتها أو نموذجها أو إدخالها فقط.

## 9. تبديل سمة المظهر

<p align="center">
  <img src="images/9-切换风格.png" alt="إعدادات المظهر مع سمات متعددة" width="840">
</p>
<p align="center"><em>الشكل 9: تصل الميزة النهائية في الإعدادات / المظهر، حيث يمكنك اختيار Pencil وDeep Night وAurora وDaylight وEmber والمزيد.</em></p>

الهدف من هذا المثال هو السماح لـ OpenWorkflows بدعم سمات مظهر متعددة. نقطة الدخول النهائية هي **الإعدادات / المظهر**.

تظهر لقطة الشاشة بطاقات سمات مثل Pencil وDeep Night وAurora وDaylight وEmber. عندما تختار واحدة، تغير الخلفية العامة واللوحات والحدود وألوان حالة التشغيل.

هذا يظهر أيضاً حالة الاستخدام الحقيقية هنا. OpenWorkflows ليس فقط للرسوم التوضيحية التجريبية. يمكنه تقسيم طلب منتج إلى بحث وتصميم وتنفيذ وتحقق وتتبع التسليم، ثم دفع كل قطعة عبر العقدة الصحيحة.

## ما أعتقد أنه مفيد بالفعل

OpenWorkflows قيم لأكثر من مجرد تغليف مطالبة في واجهة مستخدم.

إنه يربط الطلب والمخطط والبرنامج النصي والتنفيذ ومراجعة التاريخ. يمكنك توليد تدفق باللغة الطبيعية، وفحص الهيكل على اللوحة، واستخدام المطالبات الشائعة لتشديد الحدود، وعندها فقط تشغيله.

كما أن سير العمل لا يجب أن يرتبط بنموذج واحد. يمكن للعقد البسيطة استخدام نماذج أرخص، ويمكن للعقد المهمة استخدام نماذج أقوى، ويمكن أن يستمر هدف التنفيذ في التوسع إلى Claude Code وCodex وGemini أو بيئات تشغيل أخرى.

بالنسبة لمهام البرمجة المعقدة بالذكاء الاصطناعي، فإن هذا الهيكل أسهل بكثير في الصيانة من مطالبة واحدة ضخمة. إذا فشلت عقدة واحدة، أصلح تلك العقدة. إذا كان فرع واحد غير ضروري، قم بإزالته. إذا كنت تريد إعادة الاستخدام، تابع من التاريخ.

## كيف يرتبط هذا بـ Claude Code

لا يبدو OpenWorkflows كبديل لـ Claude Code.

لقد أوضح Claude Code بالفعل اتجاه سير العمل: يمكن كتابة العمل المعقد كنصوص ديناميكية، وتنسيقه عبر وكلاء فرعيين متعددين، وتشغيله في الخلفية.

يضيف OpenWorkflows طبقة مرئية إلى هذا الاتجاه: ارسم سير العمل، وحرره، واحفظه، ثم جرب نفس الهيكل ضد المزيد من النماذج وبيئات التشغيل.

لذا فهو لا يتعارض مع Claude Code. إنه يوسع فكرة سير العمل إلى الخارج.

## لا يزال مبكراً، لكنه يستحق المتابعة

OpenWorkflows ليس ناضجاً بعد. ستستمر محولات بيئات التشغيل وقدرات العقد ونظام البرامج النصية في التغير.

لكن الاتجاه واضح. لن تبقى البرمجة بالذكاء الاصطناعي عند "افتح صندوق دردشة وادفع كل خطوة يدوياً" إلى الأبد.

في النهاية، ستصبح المهام المعقدة مهام سير عمل. السؤال الحقيقي الوحيد هو ما إذا كان سير العمل هذا سيبقى محبوساً داخل أداة واحدة، أم يمكن رؤيته وتحريره ونقله وإعادة استخدامه.
