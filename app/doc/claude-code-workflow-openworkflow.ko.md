# Claude Code에 Workflow가 있다면, 다른 모델은? OpenWorkflows로 직접 해봤습니다

최근 Claude Code의 workflows를 자세히 살펴보고 있습니다.

제가 흥미롭게 생각한 점은 "또 다른 기능이 추가되었다"는 게 아니라, 복잡한 작업을 매번 채팅으로 진행하는 방식에서 벗어날 수 있다는 것입니다. 작업을 서브 에이전트, 병렬 분기, 파이프라인으로 나누고 스크립트가 이를 조정하도록 구성할 수 있습니다.

이것이 중요한 이유는, workflow가 더 이상 하나의 대화 속 임시 구성이 아니라 저장하고, 수정하고, 재사용할 수 있는 흐름이 되기 때문입니다.

그리고 저는 한 가지 질문이 생겼습니다: Workflow가 AI 코딩에서 널리 사용되는 계층이 된다면, 왜 특정 모델이나 CLI에 종속되어야 할까요?

이 질문을 따라가며 OpenWorkflows를 시험해 보았습니다. OpenWorkflows는 Claude Code 스타일 workflow를 시각적 캔버스로 만들고, 동일한 흐름을 Claude Code, Codex, Gemini 그리고 더 많은 로컬 또는 클라우드 런타임에서 사용할 수 있도록 합니다.

이번에는 추상적인 개념 없이, 스크린샷을 따라가며 설명하겠습니다. 예제도 구체적입니다: OpenWorkflows가 여러 외관 테마를 지원하도록 하고, 기본값은 Pencil로 설정하며, "설정 / 외관"에서 전환할 수 있도록 하는 것입니다.

> 스크린샷 기반의 사용 튜토리얼입니다.
>

특정 노드에 다른 모델이 필요하면 해당 노드를 선택한 뒤 노드 속성에서 모델을 덮어쓰세요. 비워 두면 workflow 또는 전역 선택을 상속합니다.

## 3. 새 Workflow를 만들고 요청 입력하기

<p align="center">
  <img src="images/3-新建workflow.png" alt="새 Workflow를 만들고 AI 입력창에 요청 입력" width="840">
</p>
<p align="center"><em>그림 3: 새 Workflow를 클릭한 후, 오른쪽 하단 AI 입력창에 workflow 내용을 설명하세요.</em></p>

일반 설정과 실행 지점을 확인한 뒤, 왼쪽에서 **새 Workflow**를 클릭하세요. 캔버스는 최소 구조(Start, Agent 하나, End)로 시작됩니다.

진짜 시작점은 수동으로 노드를 그리는 것이 아닙니다. 오른쪽 하단의 AI 입력창이 진짜 시작점입니다. 이 예제에서는 다음과 같이 입력했습니다:

```text
OpenWorkflows가 여러 외관 테마를 지원하고,
기본값은 Pencil로 하며,
설정 / 외관에서 전환할 수 있도록 해주세요.
```

`Ctrl+Enter`로 보내거나 전송 버튼을 클릭하면, OpenWorkflows가 요청을 편집 가능한 workflow 블루프린트로 변환합니다.

## 4-1. Workflow 블루프린트 생성

<p align="center">
  <img src="images/4-1生成workflow蓝图.png" alt="요청으로부터 생성된 Workflow 블루프린트" width="960">
</p>
<p align="center"><em>그림 4-1: AI가 목표를 병렬 분기, 요약 노드, 구현 노드, 검증, 전달로 분할합니다.</em></p>

요청이 전송되면, OpenWorkflows는 현재 단계를 완전한 workflow로 재작성합니다.

스크린샷의 블루프린트는 대략 다음과 같습니다:

```text
Start
  -> 외관 지원 방안 병렬 탐색
      -> 현재 진입점 조사
      -> 테마 시스템 설계
      -> Pencil 기본 테마 설계
  -> 구현 계획 요약
  -> 다중 테마 외관 구현
  -> 설정 / 외관 전환 연결
  -> 검증 및 검토
  -> 전달 결과 기록
  -> End
```

여기서 중요한 것은 그래프가 얼마나 예쁜가가 아니라, 모호한 목표가 실행 가능한 계획으로 분해된다는 점입니다.

오른쪽의 노드 속성 패널에서는 레이블, 유형, 분기, 에이전트 유형, 스키마를 계속 확인할 수 있습니다. 생성 후에도 구조 편집이 막히지 않습니다.

<p align="center">

스크린샷에서 `parallel(...)`과 `agent(...)` 구조를 볼 수 있습니다. 병렬 노드는 동시 실행 분기가 되고, 일반 노드는 개별 에이전트 호출이 됩니다.

이것은 OpenWorkflows가 단순히 상자를 그리는 도구가 아님을 보여줍니다. 캔버스 뒤에는 공유 workflow 구조가 있으며, 나중에 다른 런타임을 대상으로 지정할 수 있습니다.

## 5. 일반 프롬프트로 계속 다듬기

<p align="center">
  <img src="images/5-使用常用提示词.png" alt="일반 프롬프트 패널 및 AI 입력 영역" width="960">
</p>
<p align="center"><em>그림 5: 일반 프롬프트는 일반적인 workflow 편집 작업을 AI 입력창에 채워줍니다.</em></p>

블루프린트가 생성된 후 바로 실행할 필요는 없습니다. 오른쪽의 **일반 프롬프트** 패널이 흐름을 다듬기에 더 적합합니다.

프롬프트는 시나리오별로 그룹화되어 있습니다: 명확화, 명료성, 완전성, 비용, 구조, 신뢰성, 성능 및 병렬 처리, 검증.

스크린샷은 **요구사항 명확화** 프롬프트를 보여줍니다. 그래프를 변경하기 전에 주요 모호성을 확인하도록 AI 입력창을 채웁니다.

이것이 유용한 이유는, 많은 workflow 실패가 모델 실패가 아니기 때문입니다. 목표, 경계, 실패 경로, 비용 전략이 처음부터 충분히 명확하게 명시되지 않아서 발생합니다.

## 6. 대화형 선택으로 경계 확인하기

<p align="center">
  <img src="images/6-交互选择.png" alt="AI 응답의 대화형 선택 버튼" width="640">
</p>
<p align="center"><em>그림 6: 요청이 모호할 때 AI가 선택지를 제공하여 범위를 먼저 확인할 수 있게 합니다.</em></p>

**요구사항 명확화**를 선택한 후, AI는 그래프를 바로 변경하지 않습니다. 대신 후속 질문을 합니다: 테마 전환 기능을 어느 범위까지 구현할까요?

스크린샷은 두 가지 선택지를 제공합니다: Pencil 기본 테마만 제공하고 확장 구조를 남겨두는 것, 또는 Pencil과 여러 전환 가능한 테마를 함께 제공하는 것.

선택하면 AI가 그 결정을 workflow 블루프린트에 반영하고 업데이트된 IRGraph를 출력합니다. 이는 AI가 스스로 workflow를 잘못된 방향으로 이끌 위험을 줄여줍니다.

노드의 책임만 불분명하다면, 실행하기 전에 레이블, 프롬프트, 에이전트 유형, 스키마를 편집할 수 있습니다.

<p align="center">
  <img src="images/8-运行中.png" alt="중지 버튼이 있는 실행 중 상태" width="960">
</p>
<p align="center"><em>그림 8: 실행 중에는 버튼이 "실행 중... 중지"로 바뀌고, 각 노드가 상태를 표시합니다.</em></p>

워크플로우가 시작되면 상단 버튼이 **실행 중... 중지**로 변경됩니다. 하단의 AI 입력은 실행 중에 블루프린트가 변경되지 않도록 잠깁니다.

캔버스는 노드 상태를 직접 보여줍니다. 스크린샷에서 Start는 완료되었고, 병렬 노드는 여전히 실행 중이며, 오른쪽 상단 카운터가 실행 진행 상황을 표시합니다.

이는 긴 로그보다 훨씬 읽기 쉽습니다. 무언가 실패하면 전체 프롬프트를 버릴 필요 없이 실패한 노드를 찾아 해당 노드의 프롬프트, 모델, 입력만 조정하면 됩니다.

## 실제로 유용하다고 생각하는 점

OpenWorkflows의 가치는 단순히 프롬프트를 UI로 감싼 것에 있지 않습니다.

요구사항, 블루프린트, 스크립트, 실행, 기록 검토를 연결합니다. 자연어로 흐름을 생성하고, 캔버스에서 구조를 확인하고, 일반 프롬프트로 경계를 강화한 다음, 그때 실행할 수 있습니다.

또한 workflow가 하나의 모델에 종속될 필요도 없습니다. 간단한 노드는 저렴한 모델을, 중요한 노드는 더 강력한 모델을 사용할 수 있으며, 실행 대상은 Claude Code, Codex, Gemini 또는 다른 런타임으로 계속 확장할 수 있습니다.

복잡한 AI 코딩 작업의 경우, 이러한 구조는 거대한 단일 프롬프트보다 유지보수가 훨씬 쉽습니다. 한 노드가 실패하면 그 노드를 수정하고, 한 분기가 불필요하면 제거하고, 재사용하고 싶다면 기록에서 계속 진행하면 됩니다.

## Claude Code와의 관계

OpenWorkflows는 Claude Code를 대체하려는 것처럼 보이지 않습니다.

Claude Code는 이미 workflow 방향을 명확히 제시했습니다: 복잡한 작업을 동적 스크립트로 작성하고, 여러 서브 에이전트에 걸쳐 조정하며, 백그라운드에서 실행할 수 있습니다.

OpenWorkflows는 이 방향에 시각적 계층을 추가합니다: workflow를 그리고, 편집하고, 저장한 다음, 동일한 구조를 더 많은 모델과 런타임에 대해 시도합니다.

즉, Claude Code에 반하는 것이 아니라 Claude Code Workflow의 아이디어를 바깥으로 확장하는 것입니다.

## 아직 초기 단계지만 주목할 가치가 있습니다

OpenWorkflows는 아직 성숙하지 않았습니다. 런타임 어댑터, 노드 기능, 스크립트 생태계는 계속 변화할 것입니다.

하지만 방향은 분명합니다. AI 코딩이 "채팅창을 열고 매 단계를 수동으로 진행하는" 상태에 영원히 머물지는 않을 것입니다.

결국 복잡한 작업은 workflow가 될 것입니다. 유일한 질문은 그 workflow가 하나의 도구 안에 갇혀 있을지, 아니면 보고, 편집하고, 이전하고, 재사용할 수 있을지입니다.
