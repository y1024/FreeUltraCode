# У Claude Code есть Workflow. А как насчёт других моделей? Я попробовал OpenWorkflows

В последнее время я пристально изучаю workflow в Claude Code.

Меня интересует не просто «ещё одна функция». Она вытягивает сложную работу из одного хода чата за другим. Задачи можно разбить на подагенты, параллельные ветви и конвейеры, а затем координировать с помощью скриптов.

Это важно, потому что workflow больше не является временной договорённостью внутри одного разговора. Он становится чем-то, что можно сохранить, отредактировать и переиспользовать.

Это также подняло для меня вопрос: если workflow становятся общим слоем в AI-программировании, почему они должны быть привязаны к одной модели или одному CLI?

Поэтому я попробовал OpenWorkflows. Он превращает workflow в стиле Claude Code в визуальный холст и пытается сделать так, чтобы один и тот же workflow мог нацеливаться на Claude Code, Codex, Gemini и другие локальные или облачные среды выполнения.

Это руководство не начинается с абстрактных концепций. Оно проходит по скриншотам по порядку. Пример конкретный: сделать так, чтобы OpenWorkflows поддерживал несколько тем оформления, использовал Pencil по умолчанию и позволял переключать их в Настройки / Внешний вид.

>
> Английская версия: [OpenWorkflows usage tutorial](claude-code-workflow-openworkflow.en.md)

<p align="center">
  <img src="images/0-标题使用.png" alt="Холст OpenWorkflows, панель истории, свойства узлов и область AI-ввода" width="960">

<p align="center"><em>Рисунок 0: Основное рабочее пространство OpenWorkflows с чертежом в центре, свойствами узлов справа и AI-вводом с выводом внизу.</em></p>

Основной интерфейс состоит из четырёх частей: история workflow слева, визуальный холст в центре, свойства узлов и распространённые промпты справа, а также область AI-ввода и панели ответов внизу.

Workflow на скриншоте называется «План мультитематического оформления Pencil». Это не статическая диаграмма. Это workflow, который можно продолжать редактировать, превращать в скрипт, запускать и возвращаться к нему позже.

## 1. Скачайте OpenWorkflows

<p align="center">
  <img src="images/1-下载.png" alt="Страница OpenWorkflows на GitHub с разделом Releases" width="840">
</p>
<p align="center"><em>Рисунок 1: Найдите последнюю сборку в разделе Releases на странице GitHub.</em></p>

Самый быстрый способ попробовать — открыть страницу проекта на GitHub и скачать последний релиз из раздела Releases.

Панель About справа делает позиционирование понятным. Это визуальный редактор, который расширяет Claude Code Workflow на Codex, Gemini и другие среды выполнения LLM.

Если вы хотите работать с кодом, клонируйте репозиторий и начните из директории `app/`:

```bash
npm install
npm run dev
```

Для настольного приложения используйте:

```bash
npm run desktop
```

## 2. Проверьте общие настройки и точку запуска

<p align="center">
  <img src="images/2-通用设置.png" alt="Страница общих настроек OpenWorkflows" width="640">
</p>
<p align="center"><em>Рисунок 2: Настройте язык, локальный CLI и оболочку запуска в Настройки / Общие; активную модель / канал выбирайте внизу AI-ввода.</em></p>

Прежде чем что-либо рисовать, откройте **Настройки / Общие**. Здесь настраиваются язык интерфейса, автоперевод промптов, локальный CLI и оболочка запуска.

Старая вкладка **Модели** удалена. Активная модель или канал больше не выбираются в настройках; для текущего запроса используйте выпадающий список runtime внизу AI-ввода.

Если отдельному узлу нужна другая модель, выберите этот узел и задайте переопределение в свойствах узла. Если поле оставить пустым, узел наследует выбор workflow или глобальный выбор.

## 3. Создайте новый workflow и введите запрос

<p align="center">
  <img src="images/3-新建workflow.png" alt="Создание нового workflow и ввод запроса в AI-ввод" width="840">
</p>
<p align="center"><em>Рисунок 3: Нажмите «Новый Workflow», затем опишите workflow в AI-вводе в правом нижнем углу.</em></p>

После проверки общих настроек и точки запуска нажмите **Новый Workflow** слева. Холст начинается с минимальной структуры: Start, один Agent и End.

Настоящая отправная точка — не ручное рисование узлов. Это AI-ввод в правом нижнем углу. В этом примере я ввёл:

```text
I want OpenWorkflows to support multiple appearance themes,
default to Pencil,
and allow switching them in Settings / Appearance.
```

После отправки с помощью `Ctrl+Enter` или нажатия кнопки отправки OpenWorkflows превращает запрос в редактируемый чертёж workflow.

## 4-1. Сгенерируйте чертёж workflow

<p align="center">
  <img src="images/4-1生成workflow蓝图.png" alt="Чертёж workflow, сгенерированный из запроса" width="960">
</p>
<p align="center"><em>Рисунок 4-1: AI разбивает цель на параллельные ветви, узёл сводки, узлы реализации, валидацию и доставку.</em></p>

Как только запрос отправлен, OpenWorkflows переписывает текущий шаг в полноценный workflow.

Чертёж на скриншоте примерно становится:

```text
Start
  -> Explore appearance support in parallel
      -> Research current entry points
      -> Design the theme system
      -> Design the Pencil default theme
  -> Summarize the implementation plan
  -> Implement multi-theme appearance
  -> Connect Settings / Appearance switching
  -> Validate and review
  -> Record delivery results
  -> End
```

Важно здесь не то, насколько красиво выглядит граф. Важно то, что размытая цель становится исполняемым планом.

Панель свойств узлов справа по-прежнему позволяет просматривать метки, типы, ветви, типы агентов и схемы. Генерация не лишает вас возможности редактировать структуру.

## 4-2. Просмотрите сгенерированный скрипт

<p align="center">
  <img src="images/4-2蓝图脚本.png" alt="Диалог сгенерированного скрипта workflow" width="960">
</p>
<p align="center"><em>Рисунок 4-2: Используйте кнопку «Скрипт», чтобы просмотреть скрипт workflow, сгенерированный из холста.</em></p>

В верхней панели есть кнопка **Скрипт**. Откройте её, и вы увидите скрипт, сгенерированный из текущего чертежа.

На скриншоте видны структуры `parallel(...)` и `agent(...)`. Параллельные узлы становятся параллельными ветвями, а обычные узлы — отдельными вызовами агентов.

Это показывает, что OpenWorkflows — это не просто рисование прямоугольников. Холс поддерживается общей структурой workflow, которая впоследствии может нацеливаться на разные среды выполнения.

## 5. Продолжайте дорабатывать с помощью распространённых промптов

<p align="center">
  <img src="images/5-使用常用提示词.png" alt="Панель распространённых промптов и область AI-ввода" width="960">
</p>
<p align="center"><em>Рисунок 5: Распространённые промпты подставляют типичные правки workflow в область AI-ввода.</em></p>

После генерации чертежа не обязательно сразу запускать workflow. Панель **Распространённые промпты** справа лучше подходит для доработки процесса.

Промпты сгруппированы по сценариям: уточнение, ясность, полнота, стоимость, структура, надёжность и параллелизм, а также верификация.

На скриншоте показан промпт **Уточнить запрос**. Он заполняет AI-ввод запросом на подтверждение ключевых неоднозначностей перед изменением графа.

Это полезно, потому что многие сбои workflow — это не сбои модели. Они происходят потому, что цель, границы, пути отказа или стратегия стоимости никогда не были сформулированы достаточно чётко.

## 6. Подтвердите границы с помощью интерактивного выбора

<p align="center">
  <img src="images/6-交互选择.png" alt="Кнопки интерактивного выбора в ответе AI" width="640">
</p>
<p align="center"><em>Рисунок 6: Когда запрос неоднозначен, AI предлагает варианты, чтобы вы могли сначала подтвердить область применения.</em></p>

После выбора **Уточнить запрос** AI не меняет граф сразу. Вместо этого он задаёт уточняющий вопрос: насколько далеко должно идти переключение тем?

На скриншоте предлагается два варианта: либо выпустить только тему Pencil по умолчанию и оставить структуру расширения, либо выпустить Pencil плюс несколько переключаемых тем.

Как только вы выбираете, AI записывает это решение обратно в чертёж workflow и выводит обновлённый IRGraph. Это снижает риск того, что AI самостоятельно поведёт workflow в неправильном направлении.

## 7. Нажмите Run

<p align="center">
  <img src="images/7-运行.png" alt="Кнопка Run в верхней панели OpenWorkflows" width="960">
</p>
<p align="center"><em>Рисунок 7: После готовности чертежа нажмите Run в верхней панели.</em></p>

После подтверждения структуры, выбора runtime и ключевых границ нажмите **Run**.

Лучше не запускать workflow сразу после генерации. Сначала проверьте, имеют ли смысл параллельные ветви, находится ли узёл сводки после ветвей и охватывает ли валидация конечный результат.

Если ответственность узла неясна только по метке, вы можете отредактировать его метку, промпт, тип агента или схему перед повторным запуском.

## 8. Наблюдайте за состоянием выполнения

<p align="center">
  <img src="images/8-运行中.png" alt="Состояние выполнения с кнопкой остановки" width="960">
</p>
<p align="center"><em>Рисунок 8: Во время выполнения кнопка меняется на «Выполняется... Остановить», и каждый узел показывает своё состояние.</em></p>

Когда workflow запускается, верхняя кнопка меняется на **Выполняется... Остановить**. AI-ввод внизу блокируется, чтобы чертёж не изменился во время выполнения.

Холст напрямую показывает статус узлов. На скриншоте Start завершён, параллельный узел всё ещё выполняется, а счётчик в правом верхнем углу показывает прогресс выполнения.

Это более читаемо, чем длинный лог. Если что-то пойдёт не так, вам не нужно выбрасывать весь промпт. Вы можете найти неисправный узел и скорректировать только его промпт, модель или входные данные.

## 9. Переключите тему оформления

<p align="center">
  <img src="images/9-切换风格.png" alt="Настройки внешнего вида с несколькими темами" width="840">
</p>
<p align="center"><em>Рисунок 9: Финальная функция попадает в Настройки / Внешний вид, где можно выбрать Pencil, Deep Night, Aurora, Daylight, Ember и другие.</em></p>

Цель этого примера — сделать так, чтобы OpenWorkflows поддерживал несколько тем оформления. Конечная точка входа — **Настройки / Внешний вид**.

На скриншоте показаны карточки тем, такие как Pencil, Deep Night, Aurora, Daylight и Ember. Когда вы выбираете одну из них, меняется глобальный фон, панели, границы и цвета состояния выполнения.

Это также демонстрирует реальный сценарий использования здесь. OpenWorkflows нужен не только для демонстрационных диаграмм. Он может разбить запрос на продукт на исследование, проектирование, реализацию, валидацию и отслеживание доставки, а затем пропустить каждую часть через нужный узел.

## Что я считаю действительно полезным

OpenWorkflows ценен не только тем, что оборачивает промпт в интерфейс.

Он связывает запрос, чертёж, скрипт, выполнение и просмотр истории. Вы можете сгенерировать процесс на естественном языке, проверить структуру на холсте, использовать распространённые промпты для уточнения границ и только потом запустить его.

Workflow также не обязан быть привязан к одной модели. Простые узлы могут использовать более дешёвые модели, важные узлы — более мощные, а цель выполнения может по-прежнему расширяться на Claude Code, Codex, Gemini или другие среды выполнения.

Для сложных задач AI-программирования такая структура гораздо проще в поддержке, чем один огромный промпт. Если один узел не работает — исправьте этот узел. Если одна ветвь не нужна — удалите её. Если хотите переиспользовать — продолжите из истории.

## Как это связано с Claude Code

OpenWorkflows не выглядит заменой Claude Code.

Claude Code уже сделал направление workflow понятным: сложную работу можно записывать в виде динамических скриптов, координировать через несколько подагентов и запускать в фоновом режиме.

OpenWorkflows добавляет визуальный слой к этому направлению: рисуйте workflow, редактируйте его, сохраняйте, а затем пробуйте ту же структуру на большем количестве моделей и сред выполнения.

Так что это не противостояние Claude Code. Это расширение идеи workflow наружу.

## Пока рано, но стоит следить

OpenWorkflows ещё не зрелый. Адаптеры сред выполнения, возможности узлов и экосистема скриптов будут продолжать меняться.

Но направление ясно. AI-программирование не останется навсегда на уровне «открой чат и вручную проталкивай каждый шаг».

В конечном итоге сложные задачи станут workflow. Единственный реальный вопрос — останется ли этот workflow запертым внутри одного инструмента, или его можно будет увидеть, отредактировать, перенести и переиспользовать.
